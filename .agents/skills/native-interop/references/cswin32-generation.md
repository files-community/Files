# CsWin32 Generation In Files

## Important Files

- `src/Files.App.CsWin32/Files.App.CsWin32.csproj`: central interop project.
- `src/Files.App.CsWin32/NativeMethods.txt`: CsWin32 input list. Add Win32 functions, constants, structs, interfaces, GUID-like symbols, and wildcard families here.
- `src/Files.App.CsWin32/NativeMethods.json`: CsWin32 settings. Current project uses `allowMarshaling: false`, `public: true`, and COM `preserveSigMethods: ["*"]`.
- `src/Files.App.CsWin32/Extras.cs`: manual supplement for shapes CsWin32 does not generate cleanly, such as architecture-sensitive `SetWindowLongPtr`, extra delegates, constants, and small `LibraryImport` declarations.
- `src/Files.App.CsWin32/ManualGuid.cs`: manually exposed IID/CLSID/BHID/FOLDERID pointers using `GuidRVAGen`.
- `src/Files.App.CsWin32/IStorageProvider*.cs`: examples of custom COM structs with vtable methods generated by the repo source generator.
- `src/Files.Core.SourceGenerator/Generators/VTableFunctionGenerator.cs`: generates methods marked with `GeneratedVTableFunctionAttribute`.

## Project Model

`Files.App.CsWin32` is built for AOT-friendly interop:

- `IsTrimmable` and `IsAotCompatible` are enabled.
- `DisableRuntimeMarshalling` is enabled.
- CsWin32 is referenced with `PrivateAssets="all"`.
- `Dongle.GuidRVAGen` is used for efficient GUID pointer generation.
- The custom source generator emits explicit COM vtable calls for partial structs.

Consumers normally reference the CsWin32 project and use generated types from `Windows.Win32.*`. Existing consumers include `Files.App`, `Files.App.Storage`, and `Files.App.Server`.

## Adding A Generated API

1. Search first; the symbol may already be in `NativeMethods.txt`.
2. Add the smallest useful symbol set to `NativeMethods.txt`.
3. Build `Files.App.CsWin32`.
4. Consume the generated `PInvoke` method or type from the caller.
5. If the generator fails or emits an unexpected shape, inspect only the relevant generated output, then decide whether to adjust `NativeMethods.txt` or add a manual supplement.

Use one symbol per line. Wildcards such as `WM_*` are acceptable when the project already uses that pattern and the family is genuinely needed.

## Manual Interop Rules

Add manual declarations only when CsWin32 cannot represent the needed API, has an architecture gap, or the project needs a custom COM interface not available in metadata.

For manual P/Invoke:

- Prefer `LibraryImport` for new declarations.
- Keep declarations in `Files.App.CsWin32`, usually under the `Windows.Win32` namespace or the matching generated namespace.
- Use generated handle/value types such as `HWND`, `HANDLE`, `HRESULT`, `BOOL`, `PWSTR`, and `PCWSTR`.
- Avoid broad managed marshaling. The project disables runtime marshaling.

For manual COM:

- Model interfaces as `unsafe partial struct` types implementing `IComIID`.
- Add `static partial ref readonly Guid Guid` with `GuidRVAGen.Guid`.
- Add instance partial methods marked with `[GeneratedVTableFunction(Index = n)]`.
- Count COM vtable slots explicitly: `QueryInterface`, `AddRef`, and `Release` occupy indexes `0`, `1`, and `2`; inherited interface methods come before derived methods.
- Preserve HRESULTs and handle failures at the call site unless a local pattern intentionally throws.

## Lifetime Helpers

- `ComPtr<T>` owns a COM interface pointer and releases it on `Dispose`.
- `ComPtr<T>.Attach` takes ownership without calling `AddRef`.
- `ComPtr<T>` constructed from a pointer calls `AddRef`.
- `ComPtr<T>.GetAddressOf` is for APIs that write an output interface pointer.
- `ComHeapPtr<T>` owns memory that must be released with `CoTaskMemFree`.
- `HRESULT.ThrowIfFailedOnDebug()` throws only in Debug; production paths still need deliberate error handling.
